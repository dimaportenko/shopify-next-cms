# Product Requirements Document

## Project Name

Shopify Next CMS

## Status

Draft v0.1

## Overview

This project is a monorepo built with Turbo and `pnpm` workspaces. Its primary goal is to provide a project scaffolding command that generates a Shopify-focused Next.js application with CMS capabilities.

The intended developer experience is:

```bash
npx create-shopify-next-cms@latest my-shopify-app
```

The generated application should combine:

- A Shopify storefront powered by the Storefront GraphQL API
- A CMS editing experience powered by [Puck](https://github.com/puckeditor/puck)
- CMS content persistence in Shopify Metaobjects using the Shopify Admin API

This document captures the initial scope required to scaffold the project. Additional product and implementation details will be added later.

## Problem Statement

Developers building content-driven Shopify storefronts often need to combine:

- A custom Next.js frontend
- Shopify storefront data access
- A CMS for page composition
- A persistence layer for content that stays inside Shopify

There is no simple starter that sets up these pieces together with a single command and a sensible baseline architecture.

## Product Goal

Ship a CLI-based project generator that creates a working starter for a Shopify + Next.js + Puck CMS application, using Shopify as both the commerce backend and the CMS content storage layer.

## Primary Users

- Developers building custom Shopify storefronts
- Agencies creating reusable Shopify storefront starters
- Teams that want CMS-managed pages without introducing a separate CMS platform

## Core Value Proposition

- Fast project bootstrap with one command
- Opinionated Shopify + Next.js starter architecture
- Editable CMS pages with Puck
- CMS data stored directly in Shopify Metaobjects
- Monorepo structure that supports future packages and templates

## Scope for Initial Version

### In Scope

- Monorepo setup with Turbo and `pnpm` workspaces
- A `create-shopify-next-cms` package that scaffolds a new app
- A default Next.js template generated by the CLI
- Shopify Storefront GraphQL API integration for storefront data
- Puck integration for CMS page editing
- Shopify Admin API integration for reading and writing CMS data
- Storage of CMS configuration/content in Shopify Metaobjects
- Environment variable setup and starter documentation

### Out of Scope for Now

- Multiple starter templates
- Visual theme system beyond a basic default template
- Advanced publishing workflows
- Role-based CMS permissions beyond basic protection
- Rich media workflows and asset management
- Multi-store orchestration
- Detailed deployment automation

## Functional Requirements

### 1. Monorepo Foundation

- The repository must use Turbo for task orchestration.
- The repository must use `pnpm` workspaces for package management.
- The structure must support adding more packages and templates later.

### 2. Project Generator

- The project must expose a CLI runnable via:

```bash
npx create-shopify-next-cms@latest <project-name>
```

- The CLI must scaffold a new application into the target directory.
- The CLI must generate a usable default template without requiring manual code assembly.
- The CLI must provide basic setup guidance after generation.

### 3. Generated Next.js Template

- The generated app must be based on Next.js.
- The app must include a storefront implementation backed by Shopify Storefront GraphQL APIs.
- The app must include CMS routes and editing capabilities using Puck.
- The app must separate public storefront rendering from CMS management concerns.

### 4. Shopify Storefront Integration

- The template must be able to fetch storefront data from Shopify using GraphQL.
- The starter should support foundational storefront use cases such as homepage and catalog-driven pages.
- The integration must be configurable through environment variables.

### 5. CMS Editing with Puck

- The generated app must include Puck as the page editor.
- CMS users must be able to compose and edit page content in the app.
- Puck content/configuration must map to a format that can be stored and retrieved from Shopify.

### 6. CMS Persistence in Shopify

- CMS page data must be stored in Shopify Metaobjects.
- Read/write operations must use Shopify Admin APIs.
- The solution should define a baseline Metaobject model for CMS pages.
- The template must document required Shopify scopes and required Metaobject fields.

## Non-Functional Requirements

- The starter should be simple to understand and extend.
- The generated project should be usable as a foundation rather than a demo-only prototype.
- The architecture should favor future package reuse inside the monorepo.
- Configuration should be environment-variable driven.
- Initial setup should minimize manual Shopify-side steps where possible, while documenting unavoidable setup clearly.

## Proposed High-Level Architecture

### Monorepo

- `packages/create-shopify-next-cms`: CLI package responsible for scaffolding apps
- `templates/shopify-next-cms-default`: default application template

### Generated App Responsibilities

- Next.js app router and storefront pages
- Shopify Storefront API client layer
- Shopify Admin API client layer for CMS persistence
- Puck editor integration
- CMS API/routes for saving and loading content
- Metaobject serialization/deserialization logic

## Success Criteria for First Milestone

- A developer can run `npx create-shopify-next-cms@latest my-shopify-app`
- A project is generated successfully from the default template
- The generated app can be configured with Shopify credentials
- The storefront can fetch data from Shopify Storefront GraphQL APIs
- CMS content can be created or updated through Puck
- CMS content is persisted in Shopify Metaobjects through Admin APIs

## Assumptions

- Shopify remains the single source of truth for commerce and CMS content storage
- Puck is sufficient for the first CMS editing experience
- Metaobjects are an acceptable persistence model for structured CMS page data
- One default template is enough for the first release

## Risks and Open Questions

- Final Metaobject schema may evolve as CMS requirements become clearer
- Puck configuration format may require normalization for long-term compatibility
- Admin API authentication and route protection strategy needs more definition
- Publishing workflow, preview flow, and versioning are not yet defined
- Product requirements for page models, sections, and reusable blocks are still open

## Future Considerations

- Additional starter templates
- Theme variants
- Stronger CMS access control
- Preview and publishing workflow
- Page version history
- Shopify webhook integrations
- Reusable shared packages across generated apps

## Initial Deliverable

The first deliverable is a working scaffolded starter that proves the end-to-end flow:

1. Generate a Next.js Shopify app from the CLI
2. Connect to Shopify Storefront GraphQL APIs
3. Edit CMS content with Puck
4. Persist CMS content to Shopify Metaobjects using Admin APIs

## Notes

This PRD is intentionally lightweight and focused on the initial scaffold phase. It should be expanded as product, editor, data model, and deployment requirements become more concrete.
