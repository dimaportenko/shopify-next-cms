## Public Page Route

Published CMS pages are served at `/[slug]` via `src/app/(storefront)/[slug]/page.tsx`. This is a React Server Component that:

1. Fetches the page data from Shopify using `getCmsPageBySlug(slug)`
2. Returns `notFound()` if the page doesn't exist or isn't published
3. Wraps the page with `CmsPageShell`
4. Renders the page body using Puck's `<Render>` component

```tsx
export default async function CmsPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const page = await getCmsPageBySlug(slug);

  if (!page || page.status !== "published") {
    notFound();
  }

  return (
    <CmsPageShell page={page}>
      <Render config={puckConfig} data={page.puckData} />
    </CmsPageShell>
  );
}
```

## Shared Page Chrome

`CmsPageShell` centralizes storefront chrome for CMS-authored routes. It reads root props from the current page, checks whether the author hid the site header or footer, and then fetches shared fragments as needed.

Current behavior:

- If a published `header` fragment exists, it is rendered before the page body
- Otherwise the storefront falls back to the built-in `<Header />`
- If a published `footer` fragment exists, it is rendered after the page body
- Fragment rendering is skipped entirely when the current page sets `hideHeader: "hide"` or `hideFooter: "hide"`

This same chrome wrapper is reused inside the Puck iframe preview for non-fragment pages so the editor matches the storefront more closely.

## Contextual Rendering with `metadata`

Puck's `<Render>` component can receive extra runtime context through the `metadata` prop. Components can then read that data from `puck.metadata` inside their render functions.

This project uses that pattern for collection templates. The collection route fetches the Shopify collection and passes it into Puck:

```tsx
<Render config={puckConfig} data={page.puckData} metadata={{ collection }} />
```

The `CollectionProducts` block then reads `puck.metadata.collection` and renders the live Shopify products for the current collection handle.

The collection page editor also uses this metadata pattern for previews. Authors can set a **Preview Collection Handle** page setting in `/cms/collection/default/edit`, and the editor fetches that Shopify collection and injects it into `<Puck metadata={...} />`. That makes collection-aware blocks render against a real collection while editing, instead of falling back to the empty-state warning.

That editor-specific preview pipeline now lives in `src/app/cms/_lib/page-preview/`. The shared hook resolves preview metadata by page type, while page-specific files such as `src/app/cms/_lib/collection-preview.ts` define the root field, fetcher, and metadata mapping. Future page types like product templates can plug into the same pattern instead of duplicating editor logic inside the route.

This keeps Puck page data small and reusable while still allowing route-specific storefront data to be rendered inside CMS-authored layouts.

## Route Group: `(storefront)`

The homepage, collections, and CMS pages live inside the `(storefront)` route group. The group layout no longer renders the shared header directly. Instead, storefront routes opt into `CmsPageShell` where needed so page-level visibility settings and fragment-backed chrome can be applied consistently.

This route group still:

- Groups related routes without affecting the URL
- Separates storefront concerns from `/cms` admin routes
- Shares the root layout (theme provider, fonts) with CMS routes

## Metadata

Each CMS page generates dynamic metadata via `generateMetadata`:

```tsx
export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const page = await getCmsPageBySlug(slug);
  return { title: page?.title ?? "Page Not Found" };
}
```

## Async Params

All dynamic routes use the Next.js 15+/16 async params pattern where `params` is a `Promise` that must be awaited. This applies to both the page component and `generateMetadata`.

## Cache Revalidation

When a regular page is published via the editor, `revalidatePath` is called with the page path in the Server Action. This invalidates the Next.js cache for that specific page, ensuring the next visitor sees the updated content.

When a shared fragment is published, the CMS revalidates the root layout instead. That allows header/footer changes to propagate across storefront routes without manually revalidating every page path.
