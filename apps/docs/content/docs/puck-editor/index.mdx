The CMS uses [Puck](https://puckeditor.com) as a visual page editor. Puck is an open-source React component that provides a drag-and-drop interface for composing pages from custom components.

## Architecture

```
Editor (/cms/[pageType]/[slug]/edit)  →  Server Actions  →  Shopify Metaobjects (Admin API)
                                                          ↓
Public Pages (/[slug])     ←  <Render> (RSC)  ←  Storefront API (read)
```

- **Editor** — Client component using `<Puck>` with drag-and-drop UI
- **Editor chrome** — Puck UI is restyled to match the CMS theme using override components first, then scoped CSS for unsupported surfaces
- **CMS theming** — The admin UI supports a dedicated editor theme and a separate preview theme for the Puck iframe
- **Render** — Server component using `<Render>` for SSR performance
- **Storage** — Page content stored as JSON in a Shopify Metaobject field
- **Mutations** — Server Actions for create/update/delete (no REST API routes needed)
- **Auth** — Simple secret-based proxy (`proxy.ts`) protecting `/cms/*` routes

## Data Flow

1. **Create** — User selects a page type and title from the CMS dashboard (`/cms/new`)
2. **Edit** — Puck editor at `/cms/{pageType}/{slug}/edit` loads page data via API route and caches the client fetch with TanStack Query
3. **Publish** — `onPublish` triggers a Server Action that saves to Shopify, syncs root-level title/type metadata, revalidates the path (or the root layout for fragments), invalidates the cached editor query, and keeps the author on the current edit route
4. **View** — Public pages render through a shared `CmsPageShell` that can inject published site fragments before and after the page body

## Route Structure

```
src/app/
  layout.tsx                 # Root layout (shared by storefront + CMS)
  (storefront)/              # Route group — no URL impact
    page.tsx                 # Homepage
    [slug]/page.tsx          # Dynamic CMS pages via <Render>
  cms/
    _components/             # Private: Puck block components
      editor/                # Custom Puck editor UI overrides
    _lib/
      config.tsx             # Puck component registry + root visibility fields
      fragments.ts           # Shared fragment definitions + root prop helpers
      page-types.ts          # Page type constants + validation
      actions.ts             # Server Actions ('use server')
    cms.css                  # Scoped editor chrome theme overrides
    layout.tsx               # CMS layout (imports puck.css + cms.css)
    page.tsx                 # Dashboard — lists all pages
    login/page.tsx           # Secret-based login
    new/page.tsx             # Create page form (with type selector)
    [pageType]/[slug]/edit/page.tsx  # Puck editor
  api/
    cms/pages/[slug]/route.ts  # GET — editor data loading (?pageType= query param)
```

## Editor Customization Strategy

The CMS editor does not use Puck's default chrome as-is. The editor page applies custom overrides for the header, header actions, drawer wrapper, drawer items, fields panel, field labels, and several field types so the UI matches the rest of the CMS.

The custom header also exposes separate theme toggles for:

- **Editor theme** — changes the CMS chrome around the editor
- **Preview theme** — changes the Puck iframe preview independently from the editor chrome

When Puck exposes a stable override API, the project uses that override instead of targeting generated class names. For editor surfaces that are not yet overrideable, the project uses scoped selectors in `src/app/cms/cms.css`.

Currently the CSS fallback layer is used for:

- left rail navigation (`Blocks` / `Outline`)
- drawer section shells and group headers
- outline list selection and hover states
- viewport toolbar chrome
- preview overlay and drop zone selection / hover states

This keeps the editor visually consistent while limiting dependence on Puck internals to the smallest practical surface area.

## Shared Fragments

The CMS now supports a dedicated `fragment` page type for shared site chrome. Fragments are managed from the `/cms` dashboard and currently include:

- `header` — shared site header
- `footer` — shared site footer

Storefront routes render through `CmsPageShell`, which:

1. Reads root props from the current page
2. Checks whether `hideHeader` or `hideFooter` are set to `"hide"`
3. Fetches the corresponding published fragment pages when needed
4. Falls back to the built-in `<Header />` when no published header fragment exists

This makes header/footer content authorable in Puck without forcing every page to duplicate those blocks.
