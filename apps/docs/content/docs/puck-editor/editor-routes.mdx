## Dashboard (`/cms`)

The CMS dashboard at `/cms` is a React Server Component that directly fetches all CMS pages from Shopify. It displays each page with its title, page type badge, slug, status (draft/published), and action buttons for editing, viewing, and deleting.

## Creating Pages (`/cms/new`)

The create page form accepts a page type (General, Product, Collection, Home), title, and automatically generates a URL slug. On submit, it calls the `createPageAction` Server Action which creates a new metaobject in Shopify with empty Puck data and the selected page type, then redirects to the editor.

## Editor (`/cms/[pageType]/[slug]/edit`)

The editor page is a client component (`"use client"`) that:

1. Reads `pageType` and `slug` from route params, validates `pageType` with `isValidPageType()`
2. Fetches the page data via `GET /api/cms/pages/[slug]?pageType=[pageType]`
3. Uses TanStack Query on the client to cache the page request and handle loading/error state
4. Ensures `root.props.title` is backfilled from the saved page title when older page data does not yet include it
5. Renders the full-screen `<Puck>` editor with the page's saved data and CMS-specific UI overrides
6. On publish, calls `publishPageAction(slug, pageType, data)`, invalidates the TanStack Query cache for the current page, and stays on the same editor route instead of redirecting back to the dashboard

## Editor UI Overrides

The editor route customizes Puck in two layers:

1. **Puck overrides** in `src/app/cms/[pageType]/[slug]/edit/page.tsx`
2. **Scoped CSS theming** in `src/app/cms/cms.css` for editor surfaces without stable override hooks

Current override components live in `src/app/cms/_components/editor/` and cover:

- custom editor header and publish actions
- custom editor and preview theme toggles in the header
- custom drawer wrapper and drawer block items
- custom fields panel and field labels
- custom field renderers for `text`, `number`, `textarea`, and `select`

Root-level Puck fields now provide explicit labels such as **Title** and **Page Type**, and the custom field renderers can wrap controls in a shared label component when Puck passes label metadata.

The CSS layer is still responsible for theming Puck-managed shells and states such as:

- left rail nav (`Blocks` / `Outline`)
- drawer section cards and headers
- outline tree hover and selected states
- viewport controls
- preview overlay, selection, and drop zone highlights
- dark-mode-specific Puck surfaces that still rely on internal class names

This split is intentional: prefer stable Puck extension points where available, and use CSS only for the remaining unsupported editor chrome.

## Authentication

CMS routes are protected by Next.js 16's proxy (`src/proxy.ts`):

```ts
export function proxy(request: NextRequest) {
  if (pathname.startsWith("/cms") && !pathname.startsWith("/cms/login")) {
    const token = request.cookies.get("cms-token")?.value;
    if (token !== process.env.CMS_SECRET) {
      return NextResponse.redirect(new URL("/cms/login", request.url));
    }
  }
  return NextResponse.next();
}

export const proxyConfig = {
  matcher: ["/cms/:path*"],
};
```

The login page (`/cms/login`) sets a `cms-token` cookie when the correct `CMS_SECRET` is entered.

## Server Actions

All mutations use Server Actions defined in `src/app/cms/_lib/actions.ts`:

| Action                                      | Description                                                                                         |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `createPageAction(title, slug, pageType)`   | Creates a new CMS page metaobject with the given type                                               |
| `publishPageAction(slug, pageType, data)`   | Saves Puck data, syncs page type and title from root props when present, sets status to "published" |
| `savePageDraftAction(slug, pageType, data)` | Saves Puck data and persists the edited root title to the CMS page record                           |
| `deletePageAction(id)`                      | Deletes a CMS page metaobject                                                                       |

Each action calls `revalidatePath` to invalidate Next.js caches after mutations.

For the editor route specifically, publish also invalidates the `['cms-page', pageType, slug]` query so the editor keeps showing fresh data after save without forcing the author back to `/cms`.

## CMS Theme Modes

The CMS layout wraps editor routes in `CmsThemeProvider`, which stores two separate theme preferences in `localStorage`:

- `cms-editor-theme` — applies to the editor shell and Puck chrome
- `cms-preview-theme` — applies to the Puck iframe preview

This allows authors to edit in one theme while previewing the page in another. The preview theme is synced into the iframe document with a client-side effect and DOM observer because the preview is rendered inside a separate document.
