## Dashboard (`/cms`)

The CMS dashboard at `/cms` is a React Server Component that directly fetches all CMS pages from Shopify.

It now has two sections:

- **Site Fragments** — fixed entries for shared `header` and `footer` fragments, each with an Edit action that creates the fragment record on demand and redirects to its editor route
- **CMS Pages** — regular CMS pages with title, page type badge, slug, status (draft/published), and action buttons for editing, viewing, and deleting

## Creating Pages (`/cms/new`)

The create page form accepts a page type (General, Product, Collection, Home, Fragment), title, and automatically generates a URL slug. In practice, shared fragments are usually entered from the dashboard's **Site Fragments** section so the slug stays fixed (`header` or `footer`).

On submit, it calls the `createPageAction` Server Action which creates a new metaobject in Shopify with empty Puck data and the selected page type, then redirects to the editor.

## Editor (`/cms/[pageType]/[slug]/edit`)

The editor page is a client component (`"use client"`) that:

1. Reads `pageType` and `slug` from route params, validates `pageType` with `isValidPageType()`
2. Fetches the page data via `trpc.cms.getPageBySlug`
3. Uses TanStack Query on the client to cache the page request and handle loading/error state
4. Ensures `root.props.title` is backfilled from the saved page title when older page data does not yet include it
5. Resolves live collection preview data when editing a collection template: the root `Collection Handle` custom field searches via `trpc.cms.searchCollections`, and the selected handle is loaded through `trpc.cms.getCollectionByHandle`
6. Renders the full-screen `<Puck>` editor with the page's saved data and CMS-specific UI overrides
7. On publish, calls `publishPageAction(slug, pageType, data)`, invalidates the TanStack Query cache for the current page, and stays on the same editor route instead of redirecting back to the dashboard
8. When editing regular pages, wraps the iframe preview with the current published header/footer fragments so authors preview page chrome in context

All of those client-side reads are transported through the shared `/api/trpc` route handler.

## Editor UI Overrides

The editor route customizes Puck in two layers:

1. **Puck overrides** in `src/app/cms/[pageType]/[slug]/edit/page.tsx`
2. **Scoped CSS theming** in `src/app/cms/cms.css` for editor surfaces without stable override hooks

Current override components live in `src/app/cms/_components/editor/` and are wired together through `src/app/cms/_lib/use-editor-overrides.tsx`. They cover:

- custom editor header and publish actions
- custom editor and preview theme toggles in the header
- custom drawer wrapper and drawer block items
- custom fields panel and field labels
- custom field renderers for `text`, `number`, `textarea`, `select`, and `radio`

Root-level Puck fields now provide explicit labels such as **Title**, **Page Type**, **Collection Handle**, **Site Header**, and **Site Footer**. The collection field only appears for collection pages and uses a searchable custom picker. The last two fields are radio controls that let authors hide shared fragments on a per-page basis.

The custom field renderers can wrap controls in a shared label component when Puck passes label metadata.

The CSS layer is still responsible for theming Puck-managed shells and states such as:

- left rail nav (`Blocks` / `Outline`)
- drawer section cards and headers
- outline tree hover and selected states
- viewport controls
- preview overlay, selection, and drop zone highlights
- dark-mode-specific Puck surfaces that still rely on internal class names

This split is intentional: prefer stable Puck extension points where available, and use CSS only for the remaining unsupported editor chrome.

## Authentication

CMS routes are protected by Next.js 16's proxy (`src/proxy.ts`):

```ts
export function proxy(request: NextRequest) {
  if (pathname.startsWith("/cms") && !pathname.startsWith("/cms/login")) {
    const token = request.cookies.get("cms-token")?.value;
    if (token !== process.env.CMS_SECRET) {
      return NextResponse.redirect(new URL("/cms/login", request.url));
    }
  }
  return NextResponse.next();
}

export const proxyConfig = {
  matcher: ["/cms/:path*"],
};
```

The login page (`/cms/login`) sets a `cms-token` cookie when the correct `CMS_SECRET` is entered.

## Server Actions

All mutations use Server Actions defined in `src/app/cms/_lib/actions.ts`:

| Action                                      | Description                                                                                                    |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `createPageAction(title, slug, pageType)`   | Creates a new CMS page metaobject with the given type                                                          |
| `publishPageAction(slug, pageType, data)`   | Saves Puck data, syncs page type and title from root props when present, sets status to "published"            |
| `savePageDraftAction(slug, pageType, data)` | Saves Puck data and persists the edited root title to the CMS page record                                      |
| `deletePageAction(id)`                      | Deletes a CMS page metaobject                                                                                  |
| `editFragmentAction(formData)`              | Ensures a fixed fragment page exists for a known slug (`header`/`footer`) and redirects to `/cms/fragment/...` |

Each action calls `revalidatePath` to invalidate Next.js caches after mutations.

For the editor route specifically, publish also invalidates the current `trpc.cms.getPageBySlug` query so the editor keeps showing fresh data after save without forcing the author back to `/cms`. Publishing a regular page revalidates that page path; publishing a fragment revalidates the root layout so shared chrome updates across storefront routes.

## CMS Theme Modes

The CMS layout wraps editor routes in `CmsThemeProvider`, which stores two separate theme preferences in `localStorage`:

- `cms-editor-theme` — applies to the editor shell and Puck chrome
- `cms-preview-theme` — applies to the Puck iframe preview

This allows authors to edit in one theme while previewing the page in another. The preview theme is synced into the iframe document with a client-side effect and DOM observer because the preview is rendered inside a separate document.
