The storefront collection route lives at `apps/shopify-next-cms/src/app/(storefront)/collections/[handle]/page.tsx`.

It fetches Shopify collection data from the Storefront API, then optionally renders a published Puck page template for collection pages. If no published collection CMS page is available, the route falls back to the built-in collection product grid.

## Route Structure

Key files involved in collection rendering:

| File                                                           | Responsibility                                      |
| -------------------------------------------------------------- | --------------------------------------------------- |
| `src/app/(storefront)/collections/[handle]/page.tsx`           | Collection route: data loading + CMS render         |
| `src/components/cms-page-shell.tsx`                            | Loads shared header/footer fragments per page       |
| `src/components/cms-page-chrome.tsx`                           | Renders fragment-backed chrome with Header fallback |
| `src/components/collection-products-section.tsx`               | Fallback collection section and grid                |
| `src/components/collection-product-card.tsx`                   | Product card with image carousel                    |
| `src/app/cms/_components/blocks/collection-products-block.tsx` | Puck block that reads collection metadata           |
| `src/app/cms/_lib/config.tsx`                                  | Registers the `CollectionProducts` block            |
| `src/app/(storefront)/layout.tsx`                              | Minimal storefront route wrapper                    |
| `src/components/header.tsx`                                    | Built-in header fallback and theme controls         |
| `src/lib/shopify/queries/collections.storefront.ts`            | Collection GraphQL query                            |
| `src/lib/shopify/queries/products.storefront.ts`               | Reusable product fragment                           |
| `src/lib/shopify/queries/collections.ts`                       | Collection query-to-DTO mapping                     |
| `src/lib/shopify/queries/products.ts`                          | Reusable image/money/product mappers                |

## CMS-Backed Collection Templates

Collection routes now support a shared CMS template for `pageType: "collection"`.

The route loads two resources in parallel:

1. the Shopify collection by `handle`
2. the CMS page with `slug: "default"` and `pageType: "collection"`

If the CMS page exists and is published, the route renders it with Puck:

```tsx
const [collection, page] = await Promise.all([
  getCollection({ handle }),
  getCmsPage({ slug: "default", pageType: "collection" }).catch(() => null),
]);

return page && page.status === "published" ? (
  <Render config={puckConfig} data={page.puckData} metadata={{ collection }} />
) : (
  <CollectionProductsSection collection={collection} />
);
```

This gives you a safe rollout path:

- **published collection CMS page exists** → collection route uses the CMS layout
- **CMS page missing, draft, or fetch fails** → collection route falls back to the built-in section
- **Shopify collection missing** → route still returns `notFound()`

### Collection Context in Puck

The `CollectionProducts` Puck block reads Shopify collection data from `puck.metadata.collection` instead of storing product data inside the CMS JSON.

That keeps the CMS page reusable across all collection handles while still rendering handle-specific Shopify data at request time.

## Shared Header

Collection routes no longer get their header from the `(storefront)` layout. Instead, the collection page renders through `CmsPageShell`:

```tsx
return (
  <CmsPageShell page={page}>
    <main className="min-h-screen bg-background">
      {page && page.status === "published" ? (
        <Render
          config={puckConfig}
          data={page.puckData}
          metadata={{ collection }}
        />
      ) : (
        <CollectionProductsSection collection={collection} />
      )}
    </main>
  </CmsPageShell>
);
```

`CmsPageShell` loads the shared `header` and `footer` fragments, then `CmsPageChrome` renders them around the page body.

Current behavior:

- if a published `header` fragment exists, it is rendered before the collection content
- otherwise the storefront falls back to the built-in `<Header />`
- if a published `footer` fragment exists, it is rendered after the collection content
- page root props such as `hideHeader` and `hideFooter` can hide those shared fragments per page

The `(storefront)` layout is now only a minimal wrapper (`min-h-screen bg-background font-sans`). Shared storefront chrome lives in `CmsPageShell`, which keeps collection pages aligned with CMS-authored storefront pages and the homepage.

## Reusable Product Fragment

Collection queries no longer define product fields inline.

Instead, product-related fields are centralized in `products.storefront.ts`:

```ts
export const PRODUCT_FRAGMENT = `#graphql
  fragment ProductFragment on Product {
    id
    handle
    title
    description
    vendor
    availableForSale
    onlineStoreUrl
    tags
    featuredImage {
      url
      altText
      width
      height
    }
    images(first: 10) {
      nodes {
        url
        altText
        width
        height
      }
    }
    priceRange {
      minVariantPrice {
        amount
        currencyCode
      }
      maxVariantPrice {
        amount
        currencyCode
      }
    }
    compareAtPriceRange {
      minVariantPrice {
        amount
        currencyCode
      }
      maxVariantPrice {
        amount
        currencyCode
      }
    }
  }
`;
```

The collection query composes that fragment:

```ts
export const GET_COLLECTION_BY_HANDLE = `#graphql
  ${PRODUCT_FRAGMENT}

  query GetCollectionByHandle($handle: String!) {
    collection(handle: $handle) {
      id
      handle
      title
      description
      image {
        url
        altText
        width
        height
      }
      products(first: 12) {
        nodes {
          ...ProductFragment
        }
      }
    }
  }
`;
```

This makes the same product selection reusable in future storefront queries that return related products.

## Reusable DTOs and Mappers

Product-related DTOs were also generalized so they can be reused outside collection queries.

Defined in `src/lib/shopify/types.ts`:

```ts
export interface ImageDto {
  url: string;
  altText: string | null;
  width: number | null;
  height: number | null;
}

export interface MoneyDto {
  amount: string;
  currencyCode: string;
}

export interface ProductDto {
  id: string;
  handle: string;
  title: string;
  description: string;
  vendor: string;
  availableForSale: boolean;
  onlineStoreUrl: string | null;
  tags: string[];
  featuredImage: ImageDto | null;
  images: ImageDto[];
  priceRange: {
    minVariantPrice: MoneyDto;
    maxVariantPrice: MoneyDto;
  };
  compareAtPriceRange: {
    minVariantPrice: MoneyDto | null;
    maxVariantPrice: MoneyDto | null;
  };
}
```

Mapped in `src/lib/shopify/queries/products.ts`:

```ts
export function toProductDto(product: ProductFragmentFragment): ProductDto {
  const featuredImage = toImageDto(product.featuredImage);
  const images = product.images.nodes
    .map(toImageDto)
    .filter((image): image is ImageDto => image !== null);

  return {
    id: product.id,
    handle: product.handle,
    title: product.title,
    description: product.description,
    vendor: product.vendor,
    availableForSale: product.availableForSale,
    onlineStoreUrl: product.onlineStoreUrl ?? null,
    tags: product.tags,
    featuredImage,
    images,
    // ...price fields
  };
}
```

Collection mapping stays thin and delegates product normalization to the shared mapper.

## Collection Card Images

`collection-product-card.tsx` now prefers `product.images` for the carousel and falls back to `featuredImage` only when no gallery images are available.

```ts
function getSlides(product: ProductDto) {
  const images = product.images;

  if (images.length > 0) {
    return images;
  }

  return product.featuredImage ? [product.featuredImage] : [];
}
```

This allows collection cards to reflect real Shopify product galleries instead of repeating the same featured image.

## Maintenance Notes

When changing product fields used by collection pages:

1. Update `PRODUCT_FRAGMENT`
2. Run `pnpm --filter shopify-next-cms codegen`
3. Update shared DTOs/mappers in `products.ts`
4. Validate collection UI with:

```bash
pnpm --filter shopify-next-cms typecheck
pnpm --filter shopify-next-cms lint
```

If the UI changes, also smoke-test a collection route with Playwright, for example:

```bash
playwright-cli open http://localhost:3000/collections/gear
```
